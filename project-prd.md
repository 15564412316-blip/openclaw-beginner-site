# OpenClaw 新手安全上手站 - 产品需求文档（PRD v1.0）

## 1. 项目基本信息

### 1.1 项目名称
OpenClaw 新手安全上手站

### 1.2 项目一句话定位
一个面向零基础用户的互动式新手引导网站，帮助用户理解、安装、配置并首次使用 OpenClaw / skills，而不是让用户直接面对复杂命令行、报错日志和冷冰冰的开发文档。

### 1.3 项目目标
为零基础用户提供一个更简单、更直观、更有引导感的 OpenClaw 入门入口，降低他们从“听说过”到“真正跑起来”的门槛。

---

## 2. 项目背景

当前很多 AI 自动化工具、Agent 工具、skills 工具虽然很强，但对普通人和零基础用户不友好，主要问题包括：

1. 安装流程依赖命令行，理解门槛高
2. 遇到报错后不知道如何排查
3. 不理解 OpenClaw、skills、API Key 这些概念
4. 不知道本地安装和云端部署的区别
5. 安装完成后也不知道如何开始第一次使用
6. 官方文档和开源仓库偏技术化，不适合小白

因此，本项目不是单纯做“工具下载站”，而是做一个：
- 新手向导
- 安装路径选择器
- API Key 配置引导
- 首次任务模板中心
- 常见问题和错误诊断入口

---

## 3. 目标用户

### 3.1 核心用户
1. 零基础小白
2. 听说过 OpenClaw / Clawbot / skills，但不会安装的人
3. 想体验 AI 自动化，但不懂命令行和开发环境的人
4. 内容创作者、自由职业者、学生、办公人群
5. 希望先体验价值，再决定是否深入学习的人

### 3.2 非目标用户
1. 已经熟悉命令行和 GitHub 的高级开发者
2. 追求复杂自定义部署的技术用户
3. 需要大规模企业级自动化平台的团队

---

## 4. 用户痛点

### 4.1 安装门槛高
- 不知道终端在哪
- 看不懂命令
- 报错后不知道怎么办

### 4.2 概念门槛高
- 不知道 OpenClaw 是什么
- 不知道 skills 是什么
- 不知道 API Key 怎么获取和怎么填

### 4.3 路径不清晰
- 不知道选本地安装还是云端部署
- 不知道自己适合哪种方案
- 不知道安装后下一步做什么

### 4.4 缺少成就感
- 传统文档站太冷冰冰
- 安装过程没有反馈
- 没有明显的“完成一步”的感受

---

## 5. 产品价值主张

### 5.1 对用户的核心价值
1. 用更简单的方式理解 OpenClaw
2. 用图形化、游戏化方式替代开发者文档
3. 帮用户选对安装路径
4. 帮用户理解 API Key 接入方案
5. 帮用户在安装后快速完成第一次使用
6. 用卡片、步骤、任务的方式降低焦虑感

### 5.2 与普通文档/教程的区别
普通文档告诉用户“你应该怎么装”。

本产品要做到的是：
- 告诉用户“你现在在哪一步”
- 告诉用户“为什么要做这一步”
- 告诉用户“下一步点哪里”
- 告诉用户“如果失败了怎么办”
- 告诉用户“做完之后能得到什么”

---

## 6. 产品定位

本产品不是：
- OpenClaw 官方
- OpenClaw 本体
- 云服务平台
- 通用开发者文档站

本产品是：
- OpenClaw 新手上手入口
- 场景化安装引导站
- 零基础友好的交互式向导网站
- 后续可以扩展为本地安装器和精选 skills 市场的产品入口

---

## 7. MVP 阶段目标

### 7.1 MVP 的核心目标
在最短时间内做出一个可上线、可展示、可验证用户反馈的第一版网站。

### 7.2 MVP 要解决的问题
1. 用户能快速理解这个产品是干什么的
2. 用户能知道自己适合哪种安装方式
3. 用户能理解如何获取和配置 API Key
4. 用户能看到首次任务示例
5. 用户能通过页面留下联系方式或加入等待列表

### 7.3 MVP 不做的事情
1. 不做真正的一键本地安装器
2. 不做复杂付费系统
3. 不做完整的 skills 市场
4. 不做复杂账号体系
5. 不做自动诊断执行器
6. 不做代理充值，不代管用户 API Key

---

## 8. 产品核心功能（MVP）

## 8.1 首页
### 功能目标
让用户在 5 秒内理解：
- 这是做什么的
- 适合谁
- 为什么对我有用
- 我下一步该点哪里

### 页面内容
1. Hero 区
2. 一句话价值主张
3. 两个主按钮：
   - 开始安装
   - 先看效果
4. 3~4 个核心价值卡片
5. 简单演示区 / 流程图区
6. 常见用户类型入口

### 文案方向
- 面向零基础
- 强调简单、直观、低门槛
- 避免堆技术术语

---

## 8.2 安装路径选择器
### 功能目标
根据用户情况，推荐更适合的路径。

### 用户输入
1. 你的系统是什么？
   - macOS
   - Windows
2. 你的基础水平？
   - 完全小白
   - 能接受少量操作
   - 有一定技术基础
3. 你更在意什么？
   - 最简单
   - 更稳定
   - 数据留在本机
   - 长期在线运行

### 系统输出
推荐方案：
- 本地安装
- 云端部署
- 先体验教程，再决定

### 附带内容
- 每种方案的优缺点
- 适合人群
- 后续成本说明
- 风险和注意事项

---

## 8.3 API Key 接入中心
### 功能目标
解释什么是 API Key，以及用户如何准备自己的 Key。

### 说明内容
1. 为什么需要 API Key
2. API Key 的作用
3. 为什么本产品不直接赠送官方模型额度
4. 用户需要自己准备自己的 API Key

### 推荐接入来源
可作为文案占位，后续补链接：
- 智谱 GLM / BigModel
- OpenRouter
- SiliconFlow
- 其他后续兼容平台

### 页面要点
- 卡片式展示不同平台
- 简单说明优缺点
- 简单说明费用感知
- 提示用户自己保管自己的 Key
- 明确本产品不负责代充和托管用户 Key

---

## 8.4 首次任务模板
### 功能目标
让用户理解“安装完之后可以拿它干什么”。

### MVP 模板示例
1. 整理桌面文件
2. 总结网页内容
3. 生成内容草稿
4. 整理文本为表格
5. 自动分类文档

### 每个模板要展示的信息
- 任务名称
- 使用场景
- 适合人群
- 输入示例
- 输出示例
- 预期效果

---

## 8.5 FAQ / 帮助中心
### 功能目标
减少用户困惑和客服压力。

### 首批 FAQ 建议
1. 什么是 OpenClaw？
2. 什么是 skills？
3. 本地安装和云端部署有什么区别？
4. 为什么需要 API Key？
5. API Key 会不会泄露？
6. 我是小白，真的能用吗？
7. 安装失败怎么办？
8. 为什么我不建议一开始就买服务器？

---

## 8.6 留资 / 等待列表
### 功能目标
为后续产品验证、社群和转化做准备。

### MVP 形式
1. 邮箱收集
2. 等待列表
3. 加群/社群入口（预留）
4. 订阅更新入口

### 采集字段
- 邮箱
- 用户身份（可选）
- 当前系统（可选）
- 最想解决的问题（可选）

## 11. 设计与交互要求

### 11.1 整体风格
1. 直观
2. 游戏化
3. 卡片式
4. 不冷冰冰
5. 不要像纯开发者文档

### 11.2 视觉关键词
- 深色背景
- 渐变高亮
- 卡片布局
- 进度感
- 成就感
- 步骤引导感

### 11.3 交互原则
1. 每一步都告诉用户“现在在哪”
2. 每一步都告诉用户“下一步做什么”
3. 尽量减少长文本
4. 用模块化卡片代替大段说明
5. 页面要有明显 CTA

---

## 12. 技术实现原则

### 12.1 技术栈
- Next.js
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase（后续用于用户和数据）
- Cloudflare Pages（部署）
- Sentry（错误监控）
- Cloudflare Turnstile（防刷）

### 12.2 原则
1. 先做 MVP
2. 不过度设计
3. 页面必须响应式
4. 代码结构清晰
5. 环境变量统一管理
6. 不把密钥硬编码进代码
7. 先做低运维方案

---

## 13. 数据与账号策略

### 13.1 MVP 阶段
- 可以先不做完整登录
- 先做 waitlist 和邮箱收集

### 13.2 后续登录功能
后续如果做用户系统，需要存储：
1. 用户基本信息
2. 用户进度
3. 用户配置
4. 订单状态
5. 会员状态

### 13.3 API Key 策略
1. 用户自己准备自己的 API Key
2. 平台不代用户充值
3. 平台不承诺官方模型最低价
4. 平台尽量不托管用户明文 Key
5. 平台负责接入说明，不负责代管额度

---

## 14. 安全与运维原则

### 14.1 当前阶段重点
1. 基础表单防刷
2. 基础错误监控
3. 不暴露环境变量
4. 不保存敏感信息到前端
5. 明确用户自己保管自己的 API Key

### 14.2 当前阶段不做
1. 复杂风控
2. 自建服务器运维
3. 高复杂度权限体系
4. 复杂代理转发

---

## 15. 成功标准（MVP）

MVP 成功的判断标准：
1. 网站可以正常访问
2. 用户能理解产品定位
3. 用户能走通“查看路径 → 理解 API Key → 查看首次任务”的流程
4. 页面视觉风格不枯燥
5. 可以完成留资
6. 后续能在此基础上继续扩展

---

## 16. 开发约束

开发时遵循以下要求：

1. 所有说明尽量使用通俗中文
2. 不要一次性生成过度复杂的工程结构
3. 先实现页面，再逐步补功能
4. 优先保证目录清晰、组件复用
5. 每次改动后先检查 lint/build
6. 所有页面必须响应式
7. 所有按钮、卡片、文案都围绕“零基础用户”
8. 不要把网站做成开发者文档风格
9. 优先做首页、安装路径页、API Key 页面、首次任务页、FAQ 页
10. 每次新增代码前，先说明“为什么做这一步”
