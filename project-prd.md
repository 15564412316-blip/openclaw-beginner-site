# OpenClaw 新手安全上手站 - 产品需求文档（PRD v1.0）

## 1. 项目基本信息

### 1.1 项目名称
OpenClaw 新手安全上手站

### 1.2 项目一句话定位
一个面向零基础用户的互动式新手引导网站，帮助用户理解、安装、配置并首次使用 OpenClaw / skills，而不是让用户直接面对复杂命令行、报错日志和冷冰冰的开发文档。

### 1.3 项目目标
为零基础用户提供一个更简单、更直观、更有引导感的 OpenClaw 入门入口，降低他们从“听说过”到“真正跑起来”的门槛。

---

## 2. 项目背景

当前很多 AI 自动化工具、Agent 工具、skills 工具虽然很强，但对普通人和零基础用户不友好，主要问题包括：

1. 安装流程依赖命令行，理解门槛高
2. 遇到报错后不知道如何排查
3. 不理解 OpenClaw、skills、API Key 这些概念
4. 不知道本地安装和云端部署的区别
5. 安装完成后也不知道如何开始第一次使用
6. 官方文档和开源仓库偏技术化，不适合小白

因此，本项目不是单纯做“工具下载站”，而是做一个：
- 新手向导
- 安装路径选择器
- API Key 配置引导
- 首次任务模板中心
- 常见问题和错误诊断入口

---

## 3. 目标用户

### 3.1 核心用户
1. 零基础小白
2. 听说过 OpenClaw / Clawbot / skills，但不会安装的人
3. 想体验 AI 自动化，但不懂命令行和开发环境的人
4. 内容创作者、自由职业者、学生、办公人群
5. 希望先体验价值，再决定是否深入学习的人

### 3.2 非目标用户
1. 已经熟悉命令行和 GitHub 的高级开发者
2. 追求复杂自定义部署的技术用户
3. 需要大规模企业级自动化平台的团队

---

## 4. 用户痛点

### 4.1 安装门槛高
- 不知道终端在哪
- 看不懂命令
- 报错后不知道怎么办

### 4.2 概念门槛高
- 不知道 OpenClaw 是什么
- 不知道 skills 是什么
- 不知道 API Key 怎么获取和怎么填

### 4.3 路径不清晰
- 不知道选本地安装还是云端部署
- 不知道自己适合哪种方案
- 不知道安装后下一步做什么

### 4.4 缺少成就感
- 传统文档站太冷冰冰
- 安装过程没有反馈
- 没有明显的“完成一步”的感受

---

## 5. 产品价值主张

### 5.1 对用户的核心价值
1. 用更简单的方式理解 OpenClaw
2. 用图形化、游戏化方式替代开发者文档
3. 帮用户选对安装路径
4. 帮用户理解 API Key 接入方案
5. 帮用户在安装后快速完成第一次使用
6. 用卡片、步骤、任务的方式降低焦虑感

### 5.2 与普通文档/教程的区别
普通文档告诉用户“你应该怎么装”。

本产品要做到的是：
- 告诉用户“你现在在哪一步”
- 告诉用户“为什么要做这一步”
- 告诉用户“下一步点哪里”
- 告诉用户“如果失败了怎么办”
- 告诉用户“做完之后能得到什么”

---

## 6. 产品定位

本产品不是：
- OpenClaw 官方
- OpenClaw 本体
- 云服务平台
- 通用开发者文档站

本产品是：
- OpenClaw 新手上手入口
- 场景化安装引导站
- 零基础友好的交互式向导网站
- 后续可以扩展为本地安装器和精选 skills 市场的产品入口

---

## 7. MVP 阶段目标

### 7.1 MVP 的核心目标
在最短时间内做出一个可上线、可展示、可验证用户反馈的第一版网站。

### 7.2 MVP 要解决的问题
1. 用户能快速理解这个产品是干什么的
2. 用户能知道自己适合哪种安装方式
3. 用户能理解如何获取和配置 API Key
4. 用户能看到首次任务示例
5. 用户能通过页面留下联系方式或加入等待列表

### 7.3 MVP 不做的事情
1. 不做复杂桌面客户端安装器（先做脚本/CLI 方案）
2. 不做复杂付费系统
3. 不做完整的 skills 市场
4. 不做复杂账号体系
5. 不做“全自动自治修复型”诊断执行器（先做基础检测与人工兜底）
6. 不做代理充值，不代管用户 API Key

## 8. 产品核心功能（MVP）

## 8.1 首页
### 功能目标
让用户在 5 秒内理解：
- 这是做什么的
- 适合谁
- 为什么对我有用
- 我下一步该点哪里

### 页面内容
1. Hero 区
2. 一句话价值主张
3. 两个主按钮：
   - 开始安装
   - 先看效果
4. 3~4 个核心价值卡片
5. 简单演示区 / 流程图区
6. 常见用户类型入口

### 文案方向
- 面向零基础
- 强调简单、直观、低门槛
- 避免堆技术术语

---

## 8.2 安装路径选择器
### 功能目标
根据用户情况，推荐更适合的路径。

### 用户输入
1. 你的系统是什么？
   - macOS
   - Windows
2. 你的基础水平？
   - 完全小白
   - 能接受少量操作
   - 有一定技术基础
3. 你更在意什么？
   - 最简单
   - 更稳定
   - 数据留在本机
   - 长期在线运行

### 系统输出
推荐方案：
- 本地安装
- （可选）自动安装
- 失败后疑难支持兜底

### 附带内容
- 每种方案的优缺点
- 适合人群
- 后续成本说明
- 风险和注意事项

---

## 8.3 API Key 接入中心
### 功能目标
解释什么是 API Key，以及用户如何准备自己的 Key。

### 说明内容
1. 为什么需要 API Key
2. API Key 的作用
3. 为什么本产品不直接赠送官方模型额度
4. 用户需要自己准备自己的 API Key

### 推荐接入来源
可作为文案占位，后续补链接：
- 智谱 GLM / BigModel
- OpenRouter
- SiliconFlow
- 其他后续兼容平台

### 页面要点
- 卡片式展示不同平台
- 简单说明优缺点
- 简单说明费用感知
- 提示用户自己保管自己的 Key
- 明确本产品不负责代充和托管用户 Key

## 16. 开发约束

开发时遵循以下要求：

1. 所有说明尽量使用通俗中文
2. 不要一次性生成过度复杂的工程结构
3. 先实现页面，再逐步补功能
4. 优先保证目录清晰、组件复用
5. 每次改动后先检查 lint/build
6. 所有页面必须响应式
7. 所有按钮、卡片、文案都围绕“零基础用户”
8. 不要把网站做成开发者文档风格
9. 优先做首页、安装路径页、API Key 页面、首次任务页、FAQ 页
10. 每次新增代码前，先说明“为什么做这一步”

## 19. 部署交付与收费方案（v1）

### 19.1 目标
把网站从“教程站”升级为“部署交付入口”，核心不是让用户看完，而是让用户装成、用起来。

### 19.2 交付模式（已确认）
1. 免费模式：手动教程安装（0 元）
2. 自动模式：一键自动安装（¥99.9）
3. 一对一代办模式：付费 1v1 服务（¥199，独立购买）

### 19.3 关键产品原则
1. 不默认提供一对一远程代安装
2. 99.9 已包含安装失败协助，不做二次加价
3. 199 是独立一对一代办服务，适合不想自己动手的用户
4. 自动安装是主流程，手动教程是兜底
5. 安装完成后必须给“部署后能做什么”的任务模板入口

---

### 19.4 登录方案（v1）
#### MVP 阶段
1. 不强制登录
2. 游客可直接访问教程与安装入口
3. 仅在购买付费服务时采集必要信息（邮箱/订单号）

#### 后续阶段（可选）
1. 可增加邮箱验证码登录
2. 登录后提供订单记录、安装记录、历史诊断记录

---

### 19.5 数据存储方案（v1）
#### 技术选型
- Supabase Postgres

#### MVP 核心表
1. `waitlist_leads`：留资数据
2. `orders`：支付订单（99.9 自动安装、199 一对一代办）
3. `install_sessions`：安装任务记录（模式、系统、状态、开始/结束时间）
4. `install_checks`：步骤检测结果（步骤名、成功/失败、错误信息）
5. `support_tickets`：支持工单（99.9 与 199 用户均可提交）

#### 数据原则
1. 不存储用户明文 API Key
2. 关键表统一带时间字段（`created_at`、`updated_at`）
3. 支付回调必须幂等

---

### 19.6 收费体系方案（v1）
#### 定价（已确认）
1. 免费手动安装：0 元
2. 自动安装服务：¥99.9
3. 一对一代办服务：¥199

#### 流程设计
1. 用户先选择安装模式
2. 若选择自动安装，扫码支付 99.9 并提交订单信息，人工确认后进入自动安装流程
3. 99.9 用户安装失败可直接提交工单获取协助
4. 若用户完全不想自己操作，可直接扫码购买 199 一对一代办

#### 服务边界（明确）
1. 99.9 服务内容：自动安装脚本、自动检测、失败协助
2. 99 服务内容：一对一代办安装，不承诺“代运营/长期托管”
3. 99 可独立购买，不依赖 99.9 失败前置
4. 不提供免费人工实时一对一

#### 风险控制
1. 不把人工支持暴露为免费入口
2. 不把人工支持作为主交付模式
3. 工单按支付状态校验后才可创建

---

### 19.7 部署与技术架构（v1）
1. 前端：Next.js + Cloudflare Pages
2. 后端接口：Next.js Route Handlers
3. 数据库：Supabase
4. 支付（当前 Demo）：微信/支付宝商户收款码 + 人工确认订单
5. 支付（后续正式版）：接入聚合支付网关并做自动回调
6. 域名：Cloudflare DNS + HTTPS + 主域归一

## 20. 自动安装器技术方案（CLI v1）

### 20.1 目标
提供一个可执行安装工具，替代纯教程式引导，让用户“少操作、可验证、可回退”。

### 20.2 交付形态
1. macOS：`install.sh`
2. Windows：`install.ps1`
3. 可选全局工具：`openclaw-installer`（npm 包）

### 20.3 命令设计（建议）
1. `openclaw-installer doctor`：环境检测
2. `openclaw-installer install`：自动安装与依赖拉取
3. `openclaw-installer config`：写入/更新配置（先备份）
4. `openclaw-installer verify`：安装验证与成功标志
5. `openclaw-installer report`：导出失败日志（用于安装协助）

### 20.4 技术要求
1. 变更配置前必须自动备份（可回滚）
2. 每个步骤输出 PASS / FAIL
3. 失败必须给可复制修复命令
4. 网络异常时自动提示镜像源方案
5. 支持中断续跑（可重复执行，不破坏已有结果）
6. 产出标准化日志（JSON + 文本）

### 20.5 成功标志机制
1. Node.js 版本通过
2. Git 与 npm 可用
3. OpenClaw 安装目录存在且依赖完整
4. 配置文件校验通过
5. 首次启动命令返回成功

### 20.6 安全与合规要求
1. 不收集用户明文 API Key
2. 脚本不上传用户本地敏感文件
3. 故障上报仅采集必要日志
4. 所有下载源和命令必须可审计

### 20.7 与网站联动
1. 网站负责分流：免费 / 99.9 / 199
2. CLI 负责执行：安装、检测、回滚、报告
3. 网站展示安装状态与下一步建议
4. 自动安装失败时，99.9 用户可直接提交协助请求

### 20.8 分阶段开发
1. V1：脚本版（sh + ps1）+ 基础检测 + 日志导出
2. V2：npm CLI（跨平台）+ 中断续跑 + 错误码体系
3. V3：可选桌面 GUI 安装器（按需求再做）
